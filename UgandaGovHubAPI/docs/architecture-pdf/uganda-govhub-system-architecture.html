01 / Executive Architecture View

What the system is designed to do

Uganda GovHub API is a developer portal, sandbox, and governance console for Ministries, Departments, and Agencies. It demonstrates how government APIs can be discovered, documented, requested, approved, tested, and audited without exposing live citizen records.

Current implementation

The prototype is a local full-stack web application. The frontend is React, TypeScript, Vite, React Router, and GovHub UI components. The backend is Node.js, Express, TypeScript, OpenAPI YAML files, and PostgreSQL through the `pg` driver.

  • Routes include catalog, docs, API detail, dashboard, settings, and API registration.
  • Seeded accounts represent developer, API owner, compliance reviewer, and platform administrator roles.
  • PostgreSQL stores users, sessions, MDAs, API records, access requests, audit logs, API keys, rate limits, and verification metadata.

Government deployment intent

The architecture separates public discovery from operational access. It places policy metadata next to API contracts, requires account and API-level approval, scopes sandbox credentials, and records audit events with correlation IDs.

  • Mock records keep the showcase repeatable and safe.
  • Live NIRA, URA, URSB, MoWT, and NITA-U systems are not called in the prototype.
  • The design can be hardened for production without replacing the core governance workflow.
This submission follows the Ministry guidance by showing the main components, communication paths, storage and security controls, user access routes, APIs and integrations, hosting posture, screenshots of key workflows, and practical readiness considerations.

14 / Data Flow and Security

How data moves through the prototype

The core flow starts with API discovery and ends with an audited sandbox transaction. Access is explicit, credentials are scoped, and every allowed or denied sandbox call can be traced by correlation ID.

Page 15 of 18
1Discover
2Request
3Approve
4Audit

Stored data categories

  • Catalog and governance metadata for each MDA API product.
  • OpenAPI contracts and version records.
  • User, session, account verification, role, and MDA assignment records.
  • Access requests, lawful basis, requested fields, expiry, revocation state, and API key hashes.
  • Audit events with MDA/API context, actor, event type, correlation ID, timestamp, and redacted metadata.

Implemented security controls

  • Transport security hooks for TLS/HSTS and CORS allow-listing.
  • Password hashing, MFA support, role-based access, and verified account states.
  • AES-256-GCM sensitive-field encryption support and hash-only API key storage.
  • Scoped sandbox keys, endpoint authorization, expiry, revocation, and rate limits.
  • Server-generated correlation IDs, audit logs, and sensitive-value redaction.
No live citizen, tax, business, or permit records are exchanged. Sandbox handlers return deterministic mock responses that demonstrate interoperability behavior without exposing production government data.

Current prototype

Requests terminate inside the Express backend. Middleware validates key status and scope, then route handlers return deterministic mock responses and structured errors.

Production pathway

A production deployment would connect through approved national integration infrastructure or MDA gateways, with contract testing, adapter services, message signing, monitoring, and data-sharing agreements.

Owner accountability

Each API keeps an owning MDA, technical owner, approval authority, statutory basis, purpose limitation, retention class, and audit requirement.

External-system statement: NIRA, URA, URSB, MoWT, NITA-U, and other production government systems are not integrated in this prototype. They are represented through realistic mock contracts and sandbox handlers.

Integration control model

GovHub is positioned as a governed API portal and sandbox layer around MDA-owned services. The production version would preserve agency ownership while standardizing documentation, approval, credential control, monitoring, and audit visibility.

01MDA publishes API contract and governance metadata.
02Consumer discovers the API and submits lawful access purpose.
03Owner or reviewer approves scoped access and expiry.
04Sandbox call executes with key scope and rate-limit checks.
05Audit trail records actor, MDA, API, event, and correlation ID.

16 / Deployment Readiness

Path from showcase prototype to government deployment

The local prototype is intentionally small, but the architecture identifies where production hardening would be applied: runtime isolation, database hosting, secret management, monitoring, backup, and approved integration channels.

Page 17 of 18

Current hosting model

Frontend runs as a Vite React application. Backend runs as an Express service, defaulting to localhost. PostgreSQL and OpenAPI files support the repeatable local demonstration.

Production hardening path

Serve static assets behind approved hosting, run backend services in containers or managed application services, use managed PostgreSQL, centralized logs, backups, KMS-backed secrets, and approved integration gateways.

Scalability

The REST API is stateless apart from database and file-storage dependencies. Multiple backend instances can serve traffic behind a load balancer when shared storage is centralized.

Resilience

Production should use managed relational storage, backups, disaster recovery, health checks, deployment rollback, and separate secret management.

Security operations

Move sandbox access toward key rotation, replay protection, stronger signing, centralized logs, monitoring alerts, and tamper-evident audit anchoring where required.

RuntimeContainerized backend services behind approved ingress, health checks, and rollout controls.
DataManaged PostgreSQL, backups, retention policy, encryption keys, and recovery procedures.
SecretsKMS-backed environment secrets, key rotation, least-privilege service credentials.
ObservabilityCentral logs, metrics, request traces, audit export, and incident alerts.

Production readiness sequence

  • Replace local development servers with hosted frontend assets and load-balanced backend runtime.
  • Move OpenAPI storage and PostgreSQL persistence to managed, backed-up infrastructure.
  • Integrate approved national or MDA gateway adapters behind explicit data-sharing agreements.
  • Run contract tests, access-control tests, audit-log checks, vulnerability scanning, and operational rehearsal before pilot use.

Evaluator takeaway

The prototype is not a live national integration platform and does not exchange production citizen data. It is a working architecture model for how MDA APIs can be made discoverable, governed, documented, requestable, testable, and auditable through a common developer portal and sandbox.

The architecture is intentionally transferable: React and TypeScript on the frontend, Node.js and Express on the backend, OpenAPI as the contract format, PostgreSQL for persistence, and standard HTTP security controls for deployment hardening.
